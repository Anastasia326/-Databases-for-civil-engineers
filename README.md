Еще десять лет назад большинство инженерных данных существовало в виде отдельных файлов: Excel-таблиц, спецификаций, локальных баз семейств и папок с чертежами. 
По мере роста сложности проектов такого подхода стало недостаточно. Современный BIM-проект — это не просто набор файлов, а сложная информационная система, в которой тысячи элементов, десятки дисциплин 
и сотни участников должны работать с единым источником данных.
Чтобы понять, почему строительная отрасль постепенно переходит от файлового хранения к базам данных, необходимо разобраться, чем Excel отличается от SQL и как устроены современные хранилища информации.

# Почему Excel перестает справляться
Excel остается одним из самых популярных инструментов инженера. Он прост, понятен и позволяет быстро создавать спецификации, таблицы и отчеты.

Однако Excel создавался как электронная таблица, а не как система управления данными.

Представим жилой комплекс, содержащий:
* 15000 дверей;
* 8000 окон;
* 50000 помещений;
сотни тысяч элементов инженерных систем.

Каждый отдел ведет свои таблицы:
* архитекторы — спецификации дверей
* конструкторы — ведомости изделий
* инженеры — оборудование
* сметчики — объемы и стоимости.

Постепенно появляются десятки Excel-файлов, содержащих одни и те же данные в разных версиях.

Возникают:
* дублирование информации
* ручное обновление
* ошибки синхронизации
* потеря актуальности данных
* «информационные острова».

# Excel против SQL

Несмотря на внешнее сходство, Excel и SQL решают совершенно разные задачи.

## Структура хранения

### Excel

Информация хранится внутри файла.
``` sql
doors.xlsx
rooms.xlsx
equipment.xlsx
```
Каждый файл существует самостоятельно.
Если данные повторяются, пользователь копирует их вручную.

### SQL

Данные располагаются на сервере в виде взаимосвязанных таблиц.
```sql
Doors
Rooms
Equipment
Levels
Materials
```
Связи между таблицами позволяют избежать дублирования информации.
Предположим, имеется помещение:

| Room_ID  | Name |
| ------------- | ------------- |
| 101 | Кабинет  |

И дверь:
| Door_ID  | Room_ID |
| ------------- | ------------- |
| D-001 | 101  |

В Excel название помещения может храниться прямо в строке двери:
Если помещение переименуют, изменения придется вносить во множество строк.
В SQL хранится только ссылка: Doors.Room_ID = Rooms.Room_ID
Название помещения хранится один раз, а все двери лишь ссылаются на него.

## Масштабируемость
### Excel

Хорошо подходит для:
* небольших проектов
* локальных расчетов
* ведомостей
* временного анализа.

Однако с ростом объема данных возникают проблемы:
* большие файлы работают медленно
* увеличивается вероятность ошибок
* становится сложно поддерживать структуру.

### SQL
Базы данных проектируются для миллионов записей.

Например:
| Таблица  | Количество строк |
| ------------- | ------------- |
| Doors | 20000  |
| Windows | 15000  |
| Rooms | 7000  |
| Equipment | 200000  |

При этом скорость поиска практически не изменяется.

## Одновременная работа пользователей
### Excel

Если файл открыт одним сотрудником, остальные могут столкнуться с ограничениями:
* конфликт версий
* блокировка файла
* появление копий

### SQL

С базой одновременно могут работать:
* BIM-координаторы
* проектировщики
* аналитики
* веб-приложения
* плагины Revit
* Power BI

Все пользователи получают актуальные данные из одного источника.

* Анатомия базы данных
Несмотря на кажущуюся сложность, любая база данных состоит из нескольких базовых элементов.

## Таблица (Table)
Таблица — это набор однотипных объектов.
Например:
Rooms
| Room_ID  | Name | Area |
| ------------- | ------------- |  ------------- |
| 101 | Кабинет  | 25 |
| 102 | Коридор  | 42 |

Doors
| Door_ID  | Type | Room_ID |
| ------------- | ------------- |  ------------- |
| D-001 | ДГ-21  | 101 |

Каждая таблица отвечает за определенный тип информации.


## Поле (Field)
Поле — это столбец таблицы.

Например:
Таблица Doors содержит поля:
* Door_ID
* Type
* Width
* Height
* Room_ID
* FireRating

Каждое поле хранит один параметр объекта.

## Запись (Record)
Запись — это строка таблицы.

Например:

| Door_ID  | Type | Room_ID |
| ------------- | ------------- |  ------------- |
| D-001 | ДГ-21  | 101 |

Одна строка соответствует одному объекту.

## Первичный ключ (Primary Key)
Каждый объект должен иметь уникальный идентификатор.

Например:
Таблица помещений:
| Room_ID  | Name |
| ------------- | ------------- |
|101 | Кабинет  |
|102 | Коридор  |

Поле Room_ID является первичным ключом.

Оно:
* уникально
* минимально
* однозначно определяет запись.

В BIM аналогом может выступать ElementId или GUID элемента.

## Внешний ключ (Foreign Key)

Внешний ключ связывает таблицы между собой.
Например:
Rooms
| Room_ID  | Name |
| ------------- | ------------- |
|101 | Кабинет  |

Doors
| Door_ID  | Room_ID |
| ------------- | ------------- |
|D-001 | 101  |

Поле Room_ID в таблице Doors является внешним ключом.

Получается связь:
```
Doors
    ↓
Room_ID
    ↑
Rooms
```

Именно благодаря внешним ключам база данных превращается в единую систему, а не в набор отдельных таблиц.

## Цифровой объект как набор таблиц
Проектировщик видит дверь как трехмерный объект.
Компьютер воспринимает ее иначе.
Например, дверь содержит:
* тип
* размеры
* материал
* этаж
* помещение
* производителя

На уровне базы данных это несколько таблиц:
* Doors
* Rooms
* Levels
* Materials
* Manufacturers

Связи между ними:
```
Doors
 ↓
Room_ID → Rooms
 ↓
Level_ID → Levels
 ↓
Material_ID → Materials
```

Таким образом, BIM-модель фактически представляет собой совокупность связанных таблиц.
Сегодня информация редко хранится в одном месте.
Чаще всего она распределена между несколькими системами.

## Autodesk Construction Cloud и BIM 360
Autodesk Construction Cloud (ACC) и BIM 360 обеспечивают совместную работу над проектом.

Здесь находятся:
* модели Revit
* чертежи
* версии файлов
* документы
* замечания
* история изменений.

Сами данные не обязательно лежат в SQL-базе пользователя, однако доступны через специальные сервисы и API.
## ACC Data Connector
Data Connector позволяет выгружать данные проектов в аналитические хранилища.

Можно получать:
* перечень моделей
* информацию о документах
* данные проверок
* замечания
* версии файлов.

После этого данные могут использоваться в:
* Power BI
* SQL Server
* PostgreSQL
* корпоративных BI-системах

## Базы данных плагинов Revit

Многие компании создают собственные плагины.
Обычно они работают по схеме:

```
Revit
   ↓
Плагин
   ↓
SQLite или SQL Server
   ↓
Power BI
```

Крупные проектные институты используют корпоративные серверы данных.

Наиболее распространены:
* Microsoft SQL Server
* PostgreSQL
* Oracle Database

## Проблема информационных островов

Если каждый отдел хранит данные отдельно, возникают «информационные острова»:

```
Архитекторы
     ↓
 Excel

Конструкторы
     ↓
 Excel

ОВ и ВК
     ↓
 Excel

Сметчики
     ↓
 Excel
```

Изменения в одной таблице не попадают в другие.

В результате:
* появляются разные версии информации
* возникают ошибки в спецификациях
* увеличивается количество ручной работы
* возрастает стоимость исправлений.

Современный подход предполагает единый источник данных:

```
Архитекторы
      ↓
      SQL
 ↙     ↓      ↘
КЖ     ОВ     Смета
       ↓
    Power BI
```

# Когда все-таки Excel лучше? 
Коротко - когда мало данных.
1. Быстрый анализ и расчеты

Excel идеально подходит для разовых задач:
* подсчет объемов
* сравнение вариантов
* предварительные расчеты
* проверка результатов.

Например:
Инженеру ОВ нужно быстро оценить расход воздуховодов для нескольких вариантов трассировки.
Создавать для этого базу данных не имеет смысла — проще выполнить расчет в Excel.

Преимущества
✓ быстро
✓ не требует программирования
✓ удобно использовать формулы.

2. Небольшие объемы данных
Если данных немного, SQL будет избыточным.

Например:
Таблица оборудования на 50 позиций:
В такой ситуации Excel проще и быстрее.

Правило: до нескольких сотен или тысяч строк Excel работает вполне комфортно.

3. Отчеты и презентация данных
Даже если данные хранятся в SQL, пользователи часто выгружают их в Excel.

Причины:
* условное форматирование
* диаграммы
* сводные таблицы
* фильтрация
* привычный интерфейс.

Например, SQL выполняет запрос, а итоговый отчет инженер оформляет в Excel.
То есть:

SQL → Excel → PDF

4. Обмен информацией
Excel является фактическим стандартом обмена данными.

Производители оборудования предоставляют:
* прайс-листы
* каталоги
* спецификации
именно в формате xlsx.
Гораздо удобнее получить файл Excel, чем подключаться к чужой базе данных.

5. Индивидуальная работа
Если таблицей пользуется один человек, преимущества SQL практически исчезают.
Excel для этого вполне достаточен.


# Выводы
Excel — это удобный инструмент для работы с файлами, но не полноценная система хранения данных.
SQL-базы данных позволяют хранить информацию в виде связанных таблиц и исключают дублирование.
Основными элементами базы данных являются:
* таблицы
* поля
* записи
* первичные ключи
* внешние ключи.
BIM-модель можно рассматривать как набор взаимосвязанных таблиц.
Данные современных проектов распределены между облачными платформами, базами плагинов и корпоративными SQL-серверами.
Переход от файлового подхода к единому хранилищу данных является одним из ключевых направлений цифровизации строительной отрасли.
